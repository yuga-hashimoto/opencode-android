# OpenCode Android UI Polish to 100 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Bring the chat, runtime-not-ready, drawer, and settings screens to a reviewable 100/100 visual score through repeated API 34 emulator screenshot capture and correction.

**Architecture:** Keep the existing drawer-first navigation and screen contracts, but replace oversized Material 3 card compositions with compact, flatter OpenCode-specific components. Use the existing screenshot instrumentation test as the visual verification harness and update it only to cover real UI states, not mock layouts.

**Tech Stack:** Kotlin, Jetpack Compose Material 3, Android API 34 emulator, Android instrumentation tests, GitHub Actions.

## Global Constraints

- Work only on branch `fix/ui-cleanup-and-setup-guard` and PR #4.
- Preserve existing local runtime, remote connection, provider, agent, workspace, chat, voice, and notification behavior.
- Do not add new product functionality in this PR.
- Do not reintroduce fake sessions, fake accounts, fake plans, or placeholder schedules.
- Main chat must show only model, agent, and workspace controls; runtime selection remains inside model selection.
- Use black/graphite/white with one restrained OpenCode blue accent.
- Every visual round must pass unit tests, lint, debug/release builds, and API 34 screenshot capture.

## 100-point rubric

- Chat header and empty state: 20 points
- Composer and bottom controls: 25 points
- Runtime-not-ready recovery: 15 points
- Drawer density and hierarchy: 15 points
- Settings density and hierarchy: 15 points
- Theme consistency and verification evidence: 10 points

A screen set reaches 100 only when every rubric item below passes:

- Header height and icon sizing are compact and balanced.
- Empty state uses one small brand mark, one headline, one support line, and lightweight prompts.
- Composer is visually one compact unit and does not dominate the screen.
- Model/agent/workspace controls fit without clipping at 1080 px width.
- Runtime-not-ready state hides the unusable composer and shows two direct setup actions.
- Drawer prioritizes recent chats, uses denser rows, and keeps destinations fixed at the bottom.
- Settings uses flatter grouped rows, smaller section spacing, neutral icons, and includes wake-word access.
- No notification, ANR, permission, or system dialogs appear in evidence screenshots.
- Screenshots are real 1080x2400 API 34 emulator captures uploaded as temporary GitHub Actions artifacts; generated PNGs are not committed to Git.
